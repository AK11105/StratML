# Data Schemas and Interfaces

This document defines the **structured interfaces exchanged between Team A (Execution) and Team B (Decision)**.

Three objects cross the team boundary:

1. `DataProfile` — Team A → Team B (once, before training)
2. `ExperimentResult` — Team A → Team B (after each experiment)
3. `ActionDecision` — Team B → Team A (each iteration)

> These schemas are the **only** shared contract. Neither team reaches into the other's internals.

---

# 1. DataProfile

Generated by **Team A** after dataset ingestion. Sent to **Team B once**, before the first experiment.

Purpose: Give the decision agent dataset-level awareness so it can:
1. Decide what **preprocessing steps** are needed before training (missing value handling, encoding, scaling, imbalance correction)
2. Plan the first experiment strategy
3. Populate `StateObject.dataset_meta_features`

The agent reads this profile and encodes its preprocessing decisions into the first `ActionDecision.parameters.preprocessing` block. Team A then executes those steps before training begins.

## Schema

```
DataProfile
├── dataset_name        str
├── dataset_type        str          # "tabular" | "text" | "vision"
├── rows                int
├── columns             int
├── target_column       str
├── problem_type        str          # "classification" | "regression"
├── numerical_columns   list[str]
├── categorical_columns list[str]
├── missing_value_ratio float        # global ratio across all features
├── class_distribution  dict         # label → count (classification only)
├── feature_summary     list[FeatureInfo]
└── recommended_metrics list[str]
```

### FeatureInfo

```
FeatureInfo
├── name                str
├── dtype               str
├── unique_values       int
├── missing_percentage  float
└── distribution        str          # "normal" | "skewed" | "uniform" | "unknown"
```

### Example

```json
{
  "dataset_name": "iris",
  "dataset_type": "tabular",
  "rows": 150,
  "columns": 5,
  "target_column": "species",
  "problem_type": "classification",
  "numerical_columns": ["sepal_length", "sepal_width", "petal_length", "petal_width"],
  "categorical_columns": [],
  "missing_value_ratio": 0.0,
  "class_distribution": {"setosa": 50, "versicolor": 50, "virginica": 50},
  "recommended_metrics": ["accuracy", "f1_score"]
}
```

> Team B's `meta_features.py` derives `num_samples`, `num_features`, `feature_sample_ratio`, `class_entropy`, and `feature_variance_mean` from this object.

> The agent also uses `missing_value_ratio`, `categorical_columns`, `feature_summary.distribution`, and `class_distribution` to decide what preprocessing to apply in the first `ActionDecision`.

# 2. ExperimentResult

Generated by **Team A after each training run**. This is the primary feedback object consumed by Team B's state builder.

> `preprocessing_applied` is included so Team B knows what cleaning was already done and avoids redundant decisions in future iterations.

## Schema

```
ExperimentResult
├── experiment_id           str
├── iteration               int
├── dataset_name            str
├── model_name              str
├── model_type              str          # "ml" | "dl"
├── hyperparameters         dict
├── preprocessing_applied   PreprocessingConfig   # what was actually run before training
├── metrics                 ExperimentMetrics
├── train_curve             list[float]  # per-epoch training loss
├── validation_curve        list[float]  # per-epoch validation loss
├── runtime                 float        # seconds
├── resource_usage          ResourceUsage
└── artifacts               ArtifactRefs
```

### PreprocessingConfig

```
PreprocessingConfig
├── missing_value_strategy  str          # "mean" | "median" | "mode" | "drop"
├── scaling                 str          # "standard" | "minmax" | "none"
├── encoding                str          # "onehot" | "label" | "none"
├── imbalance_strategy      str          # "oversample" | "undersample" | "none"
└── feature_selection       str          # "variance_threshold" | "none"
```

### ExperimentMetrics

```
ExperimentMetrics
├── accuracy            float        # or None for regression
├── f1_score            float
├── precision           float
├── recall              float
├── train_loss          float        # final epoch value
├── validation_loss     float        # final epoch value
├── mse                 float        # or None for classification
├── rmse                float
└── r2                  float
```

### ResourceUsage

```
ResourceUsage
├── gpu_used            bool
├── gpu_memory_gb       float
└── cpu_time_sec        float
```

### ArtifactRefs

```
ArtifactRefs
├── model_path          str
├── metrics_file        str
└── tensorboard_logs    str
```

### Example

```json
{
  "experiment_id": "exp_003",
  "iteration": 3,
  "dataset_name": "iris",
  "model_name": "RandomForestClassifier",
  "model_type": "ml",
  "hyperparameters": {"n_estimators": 100, "max_depth": 5},
  "metrics": {
    "accuracy": 0.91,
    "f1_score": 0.90,
    "validation_loss": 0.21,
    "train_loss": 0.18
  },
  "train_curve": [0.45, 0.32, 0.22, 0.18],
  "validation_curve": [0.48, 0.35, 0.25, 0.21],
  "runtime": 4.2,
  "artifacts": {
    "model_path": "artifacts/exp_003/model.pkl",
    "metrics_file": "artifacts/exp_003/metrics.json",
    "tensorboard_logs": "runs/exp_003"
  }
}
```

# 3. ActionDecision

Generated by **Team B's policy selector** and sent to **Team A** each iteration.

Purpose: Specify exactly what the next experiment should do.

## Schema

```
ActionDecision
├── experiment_id       str          # ID for the next experiment to run
├── action_type         str          # see supported actions below
├── parameters          dict         # action-specific parameters
├── preprocessing       PreprocessingConfig   # preprocessing to apply before training
├── reason              str          # human-readable justification
├── expected_gain       float        # predicted accuracy improvement
├── expected_cost       float        # predicted runtime (seconds)
└── confidence          float        # 0.0 – 1.0
```

> `preprocessing` is always present. On the first iteration the agent derives it from `DataProfile`. On subsequent iterations it may change it (e.g. add oversampling after detecting class imbalance in results) or keep it the same.

## Supported Action Types

| action_type               | Description                                      |
|---------------------------|--------------------------------------------------|
| `switch_model`            | Replace current model with a different algorithm |
| `increase_model_capacity` | Increase layers, estimators, or hidden units     |
| `decrease_model_capacity` | Reduce model complexity to combat overfitting    |
| `modify_regularization`   | Adjust dropout, L1/L2, or weight decay           |
| `change_optimizer`        | Switch optimizer or adjust learning rate         |
| `apply_preprocessing`     | Change preprocessing strategy without switching model |
| `early_stop`              | Halt current run early based on convergence      |
| `terminate`               | End the experiment loop entirely                 |

### Example

```json
{
  "experiment_id": "exp_004",
  "action_type": "switch_model",
  "parameters": {
    "model_name": "GradientBoostingClassifier",
    "n_estimators": 200
  },
  "preprocessing": {
    "missing_value_strategy": "median",
    "scaling": "standard",
    "encoding": "onehot",
    "imbalance_strategy": "oversample",
    "feature_selection": "none"
  },
  "reason": "accuracy plateau detected over last 3 iterations",
  "expected_gain": 0.06,
  "expected_cost": 8.5,
  "confidence": 0.82
}
```

# Interface Summary

```
Team A → DataProfile    → Team B   (once, at start)
Team B → ActionDecision → Team A   (each iteration)
Team A → ExperimentResult → Team B (each iteration)
```
